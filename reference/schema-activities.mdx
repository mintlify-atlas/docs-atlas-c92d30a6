---
title: "Schema: activities"
description: "Database schema reference for the activities table, which records every food and transport emission event logged by users."
---

The `activities` table is the core data store for Carbon Tracker. Every time a user logs a food item or a transport journey, one row is inserted here. Emission totals, history views, and the weekly leaderboard are all derived from this table.

## Columns

| Column | Type | Nullable | Description |
|--------|------|----------|-------------|
| `id` | `uuid` | No | Primary key, generated by default with `gen_random_uuid()`. |
| `user_id` | `uuid` | No | Foreign key referencing `profiles.id`. Identifies the owning user. |
| `type` | `text` | No | Activity category: `"food"` or `"transport"`. |
| `name` | `text` | No | Display name of the activity (e.g. `"Beef"`, `"Car"`). |
| `emissions` | `numeric` | No | Calculated CO₂e in kilograms for this event. |
| `details` | `text` | Yes | Optional extra context stored as a JSON-serialised string (e.g. grams consumed or distance travelled). |
| `created_at` | `timestamptz` | No | Timestamp of the activity. Defaults to `now()`. Used to scope weekly rankings. |

## TypeScript interface

```typescript
export interface ActivityEntry {
  id: string
  user_id: string
  type: "food" | "transport"
  name: string
  emissions: number
  details?: string
  created_at: string
}
```

## Row-level security (RLS)

RLS is enabled on `activities`. All policies use `auth.uid() = user_id`, ensuring users can only read and modify their own activity records.

| Policy | Operation | Expression |
|--------|-----------|------------|
| Users can view their own activities | `SELECT` | `auth.uid() = user_id` |
| Users can insert their own activities | `INSERT` | `auth.uid() = user_id` |
| Users can delete their own activities | `DELETE` | `auth.uid() = user_id` |

<Warning>
  There is no UPDATE policy. Activity records are immutable after insertion. To correct a logged entry, delete it and re-insert.
</Warning>

## Query patterns

The following Supabase client queries are used throughout the app.

### Insert a new activity

```typescript
const { data, error } = await supabase
  .from("activities")
  .insert({
    user_id: userId,
    type: "food",
    name: "Beef",
    emissions: 6.75,
    details: JSON.stringify({ grams: 250 }),
  })
  .select()
  .single()
```

### Fetch all activities for the current user

```typescript
const { data, error } = await supabase
  .from("activities")
  .select("*")
  .eq("user_id", userId)
  .order("created_at", { ascending: false })
```

<Info>
  Because RLS enforces `auth.uid() = user_id`, the `.eq("user_id", userId)` filter is technically redundant but is included for clarity and to allow the query planner to use the index on `user_id`.
</Info>

### Fetch today's activities

```typescript
const today = new Date().toISOString().split("T")[0] // "YYYY-MM-DD"

const { data, error } = await supabase
  .from("activities")
  .select("*")
  .eq("user_id", userId)
  .gte("created_at", `${today}T00:00:00`)
  .order("created_at", { ascending: false })
```
